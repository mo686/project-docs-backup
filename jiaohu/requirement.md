## 1. 引言

### 1.1 背景

当前家电（洗碗机、冰箱、空调等）主控板 PCBA 在生产线上的工装测试环节使用两个独立上位机：

- **1.py（18 路电压采集上位机）**：通过 Modbus RTU 协议从硬件采集模块轮询读取 18 路测试点电压，实时显示并超阈值告警，数据存入 SQLite 数据库
- **NTC.py（6 通道 NTC 温度模拟上位机）**：通过 Modbus RTU 协议向硬件输出模块写入模拟电压，模拟 NTC 热敏电阻不同温度下的输出信号，注入被测主控板 ADC 引脚

两个上位机各自独立运行，操作人员需手动观察 1.py 回读电压后调整 NTC.py 输出参数，形成"人肉闭环"。这种方式效率低、易出错、无法支撑高节拍量产测试。

### 1.2 目标

新增联动调度引擎（AutoCoordinator），实现 NTC.py 输出与 1.py 回读之间的**温度-电压闭环自动控制**，使工装测试从半自动人工操作演进为全自动无人值守。

### 1.3 范围

- **包含**：联动调度引擎设计与实现、现有上位机改造以支持跨进程通信、闭环控制逻辑（偏差检测、自动补偿、超限联动降斜率、自动斜坡推进）、配置管理界面、联动日志
- **不包含**：现有代码缺陷修复（另案处理）、新硬件模块采购、产品外观设计、生产环境部署

### 1.4 术语与缩写

| 术语 | 说明 |
|------|------|
| DUT | Device Under Test，被测设备，即生产线上待测的家电主控板 PCBA |
| NTC | Negative Temperature Coefficient，负温度系数热敏电阻 |
| Modbus RTU | 基于 RS-485/RS-232 的串行通信协议，本项目使用功能码 03（读输入寄存器）和 06（写单保持寄存器） |
| 联动引擎 / AutoCoordinator | 新增的核心调度模块，桥接 1.py 与 NTC.py |
| 期望电压 | NTC.py 根据设定温度查曲线计算出的理论输出电压值 |
| 实际电压 / 回读电压 | 1.py 从硬件采集模块读取的真实电压值 |
| 偏差 | 实际电压 - 期望电压 |
| 斜率 | 自动模式下温度变化速率，单位 °C/min |
| 容差 | 允许的最大偏差绝对值，单位 V |
| WAL | Write-Ahead Logging，SQLite 的预写日志模式，支持一写多读并发 |

## 3. 功能需求 (FR)

### FR-01：通道绑定管理 (Priority: Must)

**描述**：用户可将 NTC.py 的 6 个输出通道与 1.py 的 18 个采集通道建立绑定关系，形成"输出-回读"对，联动引擎基于此关系执行闭环控制。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-01.1 | 系统支持创建通道绑定记录，每条记录包含：NTC 通道号(1-6)、电压采集通道号(1-18)、容差值(V)、是否启用自动补偿 |
| FR-01.2 | 同一 NTC 通道只能绑定一个电压采集通道 |
| FR-01.3 | 同一电压采集通道可被多个 NTC 通道绑定（多对一关系，用于关联通道联动） |
| FR-01.4 | 系统支持对已创建的绑定记录进行修改和删除 |
| FR-01.5 | 绑定配置持久化存储在数据库中，重启后自动加载 |
| FR-01.6 | 系统提供绑定配置编辑界面，以表格形式展示全部 6 条绑定记录 |

**验收条件**：
- 创建"CH1→CH11(容差0.05V,补偿开启)"后重启程序，配置不丢失
- 将 CH1 绑定从 CH11 改为 CH12，联动引擎下一周期生效

---

### FR-02：偏差检测 (Priority: Must)

**描述**：联动引擎在每个控制周期内，对每个已绑定的通道对执行"回读对比"：从 1.py 数据库读取最新实际电压，与 NTC.py 期望电压对比计算偏差。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-02.1 | 引擎从 `voltage_data` 表读取最新一条记录，按绑定映射取出对应通道的实际电压值 |
| FR-02.2 | 引擎从 `NTC.json` 读取当前温度设定值，结合曲线和上拉电阻计算期望电压 |
| FR-02.3 | 偏差计算公式：`偏差 = 实际电压 - 期望电压` |
| FR-02.4 | 偏差绝对值与通道容差比较，分三级判定：<br>— 偏差 ≤ 容差 → 正常，不触发动作<br>— 容差 < 偏差 ≤ 0.5V → 触发自动补偿（见 FR-03）<br>— 偏差 > 0.5V → 触发硬件故障告警 |
| FR-02.5 | 每周期偏差检测结果写入联动运行状态日志 |

**验收条件**：
- NTC CH1 设定 25°C，期望电压 2.500V，1.py CH11 回读 2.520V，偏差 0.020V，不超过容差 0.05V → 不触发动作
- NTC CH1 设定 25°C，期望电压 2.500V，1.py CH11 回读 2.600V，偏差 0.100V，超过容差 → 触发自动补偿

---

### FR-03：自动补偿 (Priority: Must)

**描述**：当偏差超过容差但未达到硬件故障级别时，联动引擎自动调整 NTC.py 输出以缩小偏差，实现闭环修正。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-03.1 | 仅对 `compensation_enabled=1` 的通道执行自动补偿 |
| FR-03.2 | 补偿量计算公式：`补偿电压 = (期望电压 - 实际电压) × 阻尼系数`<br>阻尼系数取 0.5（默认值，用户可配置范围 0.1~1.0） |
| FR-03.3 | 补偿量上限为 ±0.2V/周期，避免单周期大幅跳变 |
| FR-03.4 | 补偿通过修改 NTC.py 的当前温度设定值间接实现（反向计算所需温度差） |
| FR-03.5 | 连续补偿超过 N 次（N 可配置，默认 10 次）后偏差仍未收敛至容差以内 → 触发告警，提示可能存在非线性问题 |

**验收条件**：
- 偏差 +0.100V，阻尼 0.5 → 补偿 -0.050V，对应温度下调若干度
- 连续 3 次补偿后偏差仍在 0.08V（>容差 0.05V）→ 继续补偿
- 连续 10 次补偿后偏差仍 > 容差 → 触发告警

---

### FR-04：电压超标自动降斜率 (Priority: Must)

**描述**：当回读电压超过 1.py 中预设的告警阈值时，联动引擎自动减小对应 NTC 通道的升温斜率，防止被测板卡损坏，并在状态栏标示"(限幅)"。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-04.1 | 引擎从 `config.json` 读取 1.py 的阈值配置 |
| FR-04.2 | 回读电压 > 阈值 → 将对应通道斜率自动降低：`新斜率 = 当前斜率 × 降压系数`，降压系数默认 0.5 |
| FR-04.3 | 斜率降低后持续监控：若回读电压在 N 个周期内（默认 5 个周期）回落到阈值以下，可选择恢复原斜率 |
| FR-04.4 | 斜率降低后回读电压仍继续上升（连续 3 个周期递增）→ 再次降斜率，并触发更高级别告警 |
| FR-04.5 | 多通道同时超阈值时，每个通道独立处理 |

**验收条件**：
- 斜率 +5°C/min，CH11 回读超阈值 → 斜率自动变为 +2.5°C/min
- 降斜率后 5 个周期内回读回落 → 斜率恢复
- 降斜率后回读仍上升 → 再次降为 +1.25°C/min + 告警

---

### FR-05：通道间联动 (Priority: Should)

**描述**：当一个通道回读电压超标时，自动联动降低其他关联通道的斜率，模拟真实工况下多通道协同保护。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-05.1 | 系统支持配置通道联动规则：触发通道(NTC CHx) → 联动通道(NTC CHy) + 联动降压系数 |
| FR-05.2 | 一个触发通道可关联多个联动通道 |
| FR-05.3 | 联动条件：触发通道发生 FR-04 的"电压超标"事件后，在执行自身降斜率的同时，也对联动通道执行斜率降低 |
| FR-05.4 | 联动降斜率系数可独立配置，不必须与触发通道自身的降压系数一致 |
| FR-05.5 | 联动触发时有日志记录，标明触发通道、联动通道、降压前后斜率值 |

**验收条件**：
- 配置 CH1 超标→联动 CH2(系数 0.3)，CH1 斜率 +5→+2.5，CH2 斜率 +3→+2.1
- 日志中记录："CH1超限 → 联动降低CH2斜率: 3.0→2.1"

---

### FR-06：自动斜坡推进 (Priority: Must)

**描述**：联动引擎接管 NTC.py 的自动模式（`auto_run`）温度推进逻辑，按预设斜率逐步改变温度，在到达限温时保持，同时每个增量步都校验回读数据。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-06.1 | 仅对 `auto_run=True` 的通道执行自动斜坡 |
| FR-06.2 | 温度增量公式：`delta_T = (斜率 °C/min / 60) × 控制周期(秒)` |
| FR-06.3 | 正向斜率到达高温上限时停在限制值，负向斜率到达低温下限时停在限制值，状态标注"(限幅)" |
| FR-06.4 | 每个增量步后执行偏差检测（FR-02），确保回读与期望一致后再推进下一步 |
| FR-06.5 | 若偏差超容差，暂停推进，先执行补偿（FR-03），收敛后再继续 |
| FR-06.6 | 引擎支持暂缓/恢复自动斜坡（暂停功能） |

**验收条件**：
- 斜率 +6°C/min，上限 100°C，初始 25°C → 约 12.5 分钟后到达上限并保持
- 到达上限时状态标注"(限幅)"
- 到达上限后回读电压稳定在期望值附近（容差内）

---

### FR-07：联动配置文件管理 (Priority: Should)

**描述**：用户可保存/加载完整的联动测试配置（通道绑定 + 容差/补偿 + 联动规则 + 自动斜坡参数），形成可复用的测试方案模板。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-07.1 | 支持将当前联动配置导出为独立配置文件（JSON/INI 格式） |
| FR-07.2 | 支持导入已有的联动配置文件 |
| FR-07.3 | 配置文件包含：通道绑定、容差、补偿开关、阻尼系数、联动规则、降压系数、各通道自动斜率/限温 |
| FR-07.4 | 预置常用测试场景模板（如"冰箱全温验证"、"空调快速测试"） |

**验收条件**：
- 导出配置后，清空当前配置，再导入 → 配置完全恢复
- 选择"冰箱全温验证"模板 → 通道绑定、斜率、联动规则一键加载

### FR-08：联动状态看板 (Priority: Should)

**描述**：提供独立界面面板，实时展示联动引擎的运行状态：各通道期望/实际电压、偏差值、补偿状态、限幅状态、联动触发记录。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-08.1 | 以表格形式展示 6 个 NTC 通道的实时联动状态 |
| FR-08.2 | 表格列包含：通道名、设定温度、期望电压、实际电压、偏差、补偿状态、限幅标志、当前斜率、自动开关 |
| FR-08.3 | 偏差超过容差的通道行高亮显示（颜色分级：黄色=轻微偏差，橙色=较大偏差，红色=硬件故障级） |
| FR-08.4 | 界面实时刷新，刷新率 ≥ 5Hz |
| FR-08.5 | 支持在联动状态看板中直接修改部分参数（如临时调整容差、开关自动补偿） |

**验收条件**：
- 偏差 0.06V（容差 0.05V）→ 行显示黄色
- 偏差 0.3V → 行显示橙色
- 偏差 > 0.5V → 行显示红色 + 顶部告警横幅

---

### FR-09：联动事件日志 (Priority: Should)

**描述**：联动引擎产生的所有关键事件（偏差告警、限幅触发、补偿执行、超限降斜率、联动触发、大偏差告警）均记录到数据库并支持查询导出。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-09.1 | 事件日志包含字段：时间戳、事件类型、事件描述、详细信息（JSON） |
| FR-09.2 | 事件类型枚举：`DEV_ALARM`(偏差告警)、`LIMIT_HIT`(限幅)、`COMPENSATION`(补偿)、`SLOPE_ADJ`(降斜率)、`LINK_TRIGGER`(联动触发)、`HW_FAULT`(硬件故障)、`RAMP_COMPLETE`(斜坡完成)、`AUTO_STOP`(自动停止) |
| FR-09.3 | 事件实时追加写入数据库，同时在前端联动日志面板中展示 |
| FR-09.4 | 支持按事件类型、通道号、时间范围筛选查询 |
| FR-09.5 | 支持导出事件日志为 Excel |

**验收条件**：
- 通道发生补偿事件 → 日志立即出现一条 `COMPENSATION` 记录
- 筛选"CH1 + DEV_ALARM + 今日" → 仅返回匹配记录

### FR-10：启动/停止全自动测试 (Priority: Must)

**描述**：用户配置好通道绑定和联动参数后，一键启动全自动测试；测试过程中可暂停/停止。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-10.1 | 提供"启动全自动"按钮，点击后联动引擎接管所有已绑定通道的控制 |
| FR-10.2 | 启动前执行自检：验证 NTC.py 和 1.py 均处于连接状态、通道绑定完整性、数据库可访问 |
| FR-10.3 | 自检失败时明确提示失败原因，阻止启动 |
| FR-10.4 | 提供"暂停"按钮，暂停后所有通道保持当前输出不变 |
| FR-10.5 | 提供"停止"按钮，停止后所有通道输出归零/复位 |
| FR-10.6 | 异常情况（串口断开、长时间无数据、大偏差）下自动停止并告警 |
| FR-10.7 | 正常完成全温度扫描后自动停止并通知用户 |

**验收条件**：
- NTC.py 未连接 → 点击启动 → 提示"NTC.py 未连接，无法启动"
- 启动后关掉 NTC.py → 30 秒内引擎检测到异常 → 自动停止 + 告警
- 全部通道完成上限扫描 → 自动停止 + 通知"全自动测试完成"

### FR-11：两个上位机状态检测 (Priority: Must)

**描述**：联动引擎需能检测 NTC.py 和 1.py 的运行状态（是否在运行、串口是否连接、数据是否在更新）。

**详细需求**：

| 编号 | 需求描述 |
|------|----------|
| FR-11.1 | 通过 `NTC.json` 的 `_heartbeat` 字段或文件修改时间判断 NTC.py 是否在运行 |
| FR-11.2 | 通过 `voltage_data` 表最新记录的时间戳判断 1.py 数据是否在更新 |
| FR-11.3 | 任一上位机超过超时阈值（默认 10 秒）无更新 → 判定为异常 |
| FR-11.4 | 异常状态在联动状态看板中醒目提示 |

**验收条件**：
- 停止 1.py 采集 → 10 秒后引擎判定"1.py 数据停止更新"
- NTC.py 崩溃退出 → 10 秒后引擎判定"NTC.py 无心跳"

## 4. 非功能需求 (NFR)

### NFR-01：性能需求 (Priority: Must)

| 编号 | 需求描述 |
|------|----------|
| NFR-01.1 | 控制周期 ≤ 200ms（即每秒至少执行 5 次完整的闭环控制循环） |
| NFR-01.2 | 偏差检测 + 补偿计算延迟 ≤ 50ms |
| NFR-01.3 | 联动状态看板刷新率 ≥ 5Hz |
| NFR-01.4 | 数据库写入不阻塞控制循环（异步批量写入） |
| NFR-01.5 | 内存占用增量 ≤ 100MB（相对未运行联动引擎状态） |

**验收条件**：
- 用时间戳打点验证连续 100 个控制周期的平均耗时 < 200ms
- 状态看板界面肉眼观察无明显卡顿（刷新间隔 ≤ 200ms）

---

### NFR-02：稳定性需求 (Priority: Must)

| 编号 | 需求描述 |
|------|----------|
| NFR-02.1 | 联动引擎支持连续运行 ≥ 24 小时无崩溃、无内存泄漏 |
| NFR-02.2 | 串口断开后自动重连机制，重连间隔 1-5 秒，重连期间控制循环标记异常但不崩溃 |
| NFR-02.3 | NTC.py 或 1.py 异常退出后联动引擎保持运行，等待上位机重新启动后自动恢复联动 |
| NFR-02.4 | JSON 配置文件写入使用临时文件+原子重命名的安全写入方式，断电不丢配置 |
| NFR-02.5 | 控制循环异常(如单周期计算错误)不影响下一周期，错误记录日志后继续 |

**验收条件**：
- 挂机运行 24h，内存占用无持续增长趋势
- 运行中拔掉串口 → 等待 5 秒 → 插回 → 自动恢复

---

### NFR-03：兼容性需求 (Priority: Should)

| 编号 | 需求描述 |
|------|----------|
| NFR-03.1 | 支持 Windows 10/11 操作系统（32/64 位均兼容） |
| NFR-03.2 | Python 3.8 及以上版本 |
| NFR-03.3 | 与现有 1.py 和 NTC.py 的配置文件格式向后兼容，不改动其现有字段结构 |
| NFR-03.4 | 联动引擎不能影响 1.py 和 NTC.py 各自的独立运行（引擎未启动时照样正常工作） |

**验收条件**：
- 不启动联动引擎，直接运行 1.py 和 NTC.py → 功能与改造前完全一致
- 联动引擎读取 `NTC.json` 后文件结构不变，NTC.py 正常读写不受影响

### NFR-04：可维护性需求 (Priority: Should)

| 编号 | 需求描述 |
|------|----------|

### IR-02：NTC.json 扩展字段 (Priority: Must)

在现有 `NTC.json` 中新增以下字段，不破坏现有字段结构：

```json
{
  "... 现有字段保持不变 ...": "",
  "_heartbeat": {
    "timestamp": 1234567890.123,
    "pid": 12345
  },
  "_external_command": {
    "0": { "temp": 25.5, "slope": 3.0, "auto_run": true },
    "1": { "slope": 2.0 }
  }
}
```

| 字段 | 说明 |
|------|------|
| `_heartbeat.timestamp` | NTC.py 每次控制周期写入的当前时间戳，联动引擎用于检测 NTC.py 存活 |
| `_heartbeat.pid` | NTC.py 进程 ID，辅助判断进程重启 |
| `_external_command` | 联动引擎向 NTC.py 下发的控制指令，key 为通道序号(0-based string)，value 包含待修改的参数。NTC.py 执行后清空该字段 |

---

### IR-03：NTC.py 控制指令通道 (Priority: Must)

联动引擎通过 `_external_command` 下发以下控制指令：

| 指令字段 | 类型 | 范围 | 说明 |
|----------|------|------|------|
| temp | float | -50.0 ~ 150.0 | 修改指定通道的温度设定值 |
| slope | float | -600 ~ 600 | 修改指定通道的斜率 |
| auto_run | bool | true/false | 启用/禁用自动模式 |

NTC.py 需新增定时器（建议 200ms 周期）轮询 `NTC.json` 中的 `_external_command` 字段，检测到指令后执行并清空。

---

### IR-04：config.json 读取 (Priority: Must)

联动引擎需读取 `config.json` 中的 `thresholds` 字段（18 路电压告警阈值），用于 FR-04 超阈值判断。读取为只读方式，不修改该文件。

## 6. 数据需求

### 6.1 控制数据流

```
┌─────────────────┐
│   NTC.json      │  读取 NTC 状态
│  (NTC状态)      │  ┌──────────────────────┐
└─────────────────┘  │                      │
                     │   AutoCoordinator     │
┌─────────────────┐  │                      │  DB读写
│  voltage_data   │  │   1. 读期望(温度→电压)│──────→ DB写入(状态/日志)
│   .db           │──│   2. 读实际(采集电压)  │
│  (1.py实时数据) │  │   3. 对比偏差          │
└─────────────────┘  │   4. 补偿/限幅/联动   │
                     │   5. 更新NTC温度       │
┌─────────────────┐  │   6. 写日志            │
│  config.json    │  └──────────┬───────────┘
│  (阈值配置)     │── 读取阈值   │
└─────────────────┘             │ 写入控制指令
                     ┌──────────▼───────────┐
                     │   NTC.json           │
                     │  (_external_command) │
                     └──────────────────────┘
```

### 6.2 数据持久化策略

| 数据 | 存储位置 | 写入策略 | 保留周期 |
|------|----------|----------|----------|
| 通道绑定配置 | voltage_data.db / channel_mapping | 用户修改时即时写入 | 永久 |
| 联动规则 | voltage_data.db / link_rules | 用户修改时即时写入 | 永久 |
| 联动运行状态 | voltage_data.db / link_run_state | 每控制周期写入一条 | 保留最近 7 天，自动清理 |
| 联动事件日志 | voltage_data.db / link_events | 事件发生时即时写入 | 永久（用户手动清理） |
| 联动配置模板 | 独立 JSON 文件 | 用户导出/导入 | 永久 |

### 8.2 集成验收测试场景

**场景一：冰箱主控板标准测试**

1. 配置绑定：NTC CH1→1.py CH11, NTC CH2→1.py CH12, NTC CH3→1.py CH13
2. 配置联动：CH1超限→联动CH2(系数0.4)
3. 设置：CH1(冷藏) 0~5℃(+1℃/min), CH2(冷冻) -18~-12℃(-2℃/min), CH3(环温) 10~40℃(+3℃/min)
4. 启动全自动
5. 验收：三个通道按各自斜率推进到限温，偏差均在容差内，联动日志无异常告警

**场景二：异常模拟测试**

1. CH1 回读电压故意制造 0.3V 偏差
2. 验收：引擎触发自动补偿，补偿记录出现在日志中
3. CH2 回读电压超过阈值
4. 验收：CH2 斜率自动降为设定值的 50%，状态标注"(限幅)"

**场景三：稳定性测试**

1. 全自动模式运行 24 小时
2. 验收：无崩溃、无内存泄漏、日志完整、配置文件无损坏
